# 모임 총무 관리 시스템 화면 요구사항 및 기능 명세서

## 1. 문서 개요
- 문서명: 모임 총무 관리 시스템 화면 요구사항 및 기능 명세서
- 작성일: 2026-03-08
- 기준 문서: `모임총무_관리시스템_기획서.md`
- 대상 버전: MVP 1.0
- 목적: 총무 전용 회비/지출 관리 도구의 화면 요구사항과 기능 명세를 개발 가능한 수준으로 구체화한다.

## 2. 범위
본 문서는 다음 범위를 포함한다.

- 총무가 사용하는 관리 화면
- 회비, 입금, 지출, 정산, 월 마감 관련 핵심 기능
- 기초잔액, 이월금, 변경 이력, 첨부 파일 관리 규칙
- 검증, 계산, 예외 처리 기준

본 문서는 다음 범위를 제외한다.

- 대외 공개용 조회 기능
- 외부 공유 링크 기능
- 다중 승인 워크플로우
- 은행 실시간 API 연동
- 세무 처리 기능

## 3. 사용자 정의
### 3.1 총무
- 회원 등록, 수정, 비활성화 가능
- 회비 항목 생성, 수정 가능
- 납부 상태 변경 가능
- 입금 등록 및 예외 처리 가능
- 지출 등록, 수정, 삭제 가능
- 기초잔액 설정, 월 마감, 변경 이력 조회 가능

## 4. 공통 요구사항
### 4.1 공통 UI/UX 요구사항
- 모바일과 데스크톱에서 모두 사용 가능해야 한다.
- 금액은 원화 기준 천 단위 구분자를 표시한다.
- 날짜는 `YYYY-MM-DD` 형식을 기본으로 사용한다.
- 목록 화면은 검색, 정렬, 필터 기능을 제공한다.
- 저장 성공/실패 결과는 즉시 알려야 한다.
- 삭제 기능은 확인 절차를 거쳐야 한다.

### 4.2 공통 접근 요구사항
- 인증된 총무만 관리 화면에 접근할 수 있어야 한다.
- 서버와 화면 모두 권한 없는 요청을 차단해야 한다.
- 모든 데이터 변경 행위는 이력으로 남길 수 있어야 한다.

### 4.3 공통 데이터 처리 요구사항
- 수입과 지출은 모두 거래 단위로 기록되어야 한다.
- 거래 등록 후 대시보드와 정산 수치는 즉시 반영되어야 한다.
- 필수 입력값이 누락되면 저장할 수 없어야 한다.
- 금액, 날짜, 중복 입력 가능성은 검증해야 한다.
- 잔액 계산에는 기초잔액 또는 직전 마감 이월금을 반영해야 한다.
- 마감된 기간의 거래는 기본적으로 직접 수정할 수 없어야 한다.

## 5. 화면별 상세 요구사항

## 5.1 S-01 로그인 화면
### 목적
- 총무만 시스템에 접근하도록 한다.

### 대상 사용자
- 총무

### 주요 구성 요소
- 아이디 또는 이메일 입력
- 비밀번호 입력
- 로그인 버튼
- 로그인 오류 메시지

### 주요 액션
- 로그인
- 로그아웃

### 상세 요구사항
- 올바른 계정 정보 입력 시 대시보드로 이동해야 한다.
- 잘못된 정보 입력 시 오류 메시지를 표시해야 한다.
- 로그인 상태는 일정 시간 유지되어야 한다.

### 예외 처리
- 연속 로그인 실패 시 일시 잠금 또는 추가 확인 절차를 적용할 수 있다.
- 인증되지 않은 사용자가 관리 URL로 접근하면 로그인 화면으로 이동시킨다.

## 5.2 S-02 대시보드 화면
### 목적
- 총무가 현재 회계 상태와 처리 우선순위를 한눈에 확인하도록 한다.

### 대상 사용자
- 총무

### 주요 구성 요소
- 기초잔액 카드
- 현재 잔액 카드
- 이번 달 총 수입 카드
- 이번 달 총 지출 카드
- 미납 인원 카드
- 확인 필요 입금 건수 카드
- 최근 입금 내역 목록
- 최근 지출 내역 목록
- 기간 선택 필터

### 표시 항목
- 현재 조회 기간의 기초잔액
- 현재 잔액
- 기간 수입 합계
- 기간 지출 합계
- 미납 회원 수
- `확인 필요` 상태 입금 건수
- 최근 5건 입금 정보
- 최근 5건 지출 정보

### 주요 액션
- 기간 변경
- 미납 카드 클릭 시 납부 현황 화면 이동
- 확인 필요 카드 클릭 시 입금 처리 화면 이동
- 최근 거래 항목 클릭 시 상세 화면 이동

### 상세 요구사항
- 기본 조회 기간은 현재 월로 설정한다.
- 기간 변경 시 카드와 목록 데이터가 함께 갱신되어야 한다.
- 잔액이 음수인 경우 경고 스타일을 표시해야 한다.
- 아직 월 마감이 안 된 경우 마감 필요 상태를 표시할 수 있어야 한다.

### 예외 처리
- 거래 데이터가 없을 경우 `데이터 없음` 상태를 표시해야 한다.
- 기초잔액이 설정되지 않았을 경우 설정 유도 메시지를 보여줘야 한다.

## 5.3 S-03 회원 목록 화면
### 목적
- 회비 대상 회원을 검색, 조회, 관리한다.

### 대상 사용자
- 총무

### 주요 구성 요소
- 회원 검색창
- 상태 필터
- 회비 대상 여부 필터
- 회원 목록 테이블
- 회원 등록 버튼

### 목록 표시 항목
- 이름
- 닉네임
- 연락처
- 입금자명
- 회비 대상 여부
- 상태
- 최근 납부 상태 요약

### 주요 액션
- 회원 검색
- 상태 필터
- 회원 등록
- 회원 선택 후 수정

### 상세 요구사항
- 이름, 닉네임, 입금자명 기준 검색이 가능해야 한다.
- 활성/비활성 회원 필터가 가능해야 한다.
- 회비 대상 여부 필터가 가능해야 한다.
- 동일 이름 회원이 여러 명이면 보조 식별 정보를 함께 보여줘야 한다.

### 예외 처리
- 검색 결과가 없으면 빈 상태 메시지를 표시해야 한다.

## 5.4 S-04 회원 등록/수정 화면
### 목적
- 회원 정보를 생성하거나 수정한다.

### 대상 사용자
- 총무

### 입력 항목
- 이름
- 닉네임
- 연락처
- 입금자명
- 회비 대상 여부
- 상태
- 메모

### 필수 입력값
- 이름
- 상태

### 주요 액션
- 저장
- 취소
- 비활성화

### 상세 요구사항
- 신규 회원 등록 시 기본 상태는 `활성`으로 설정한다.
- 입금자명이 회원 이름과 다를 수 있으므로 별도 입력을 허용한다.
- 회비 대상 여부를 끄면 신규 회비 자동 대상에서 제외한다.
- 회원 비활성화 시 기존 거래 이력은 유지되어야 한다.

### 검증 규칙
- 연락처는 숫자와 하이픈 형식만 허용한다.
- 이름이 비어 있으면 저장할 수 없다.
- 동일 이름 등록은 허용하되 경고를 표시할 수 있다.

## 5.5 S-05 회비 항목 목록 화면
### 목적
- 생성된 회비 항목을 조회하고 상태를 관리한다.

### 대상 사용자

### 주요 구성 요소
- 회비 항목 목록
- 기간 필터
- 상태 필터
- 회비 생성 버튼

### 목록 표시 항목
- 회비명
- 회비 구분
- 금액
- 대상 기간
- 마감일
- 대상 인원 수
- 납부 완료 인원 수
- 미납 인원 수
- 확인 필요 인원 수

### 주요 액션
- 회비 생성
- 회비 선택 후 납부 현황 이동
- 상태 필터 적용

### 상세 요구사항
- 현재 진행 중인 회비 항목이 상단에 우선 노출되어야 한다.
- 마감일이 지난 항목은 상태를 시각적으로 구분해야 한다.
- 목록에서 즉시 미납 규모를 파악할 수 있어야 한다.

## 5.6 S-06 회비 항목 생성/수정 화면
### 목적
- 월회비, 분기회비, 행사비 등 회비 항목을 생성한다.

### 대상 사용자
- 총무

### 입력 항목
- 회비명
- 회비 구분
- 금액
- 대상 기간
- 마감일
- 대상 회원 범위
- 설명

### 필수 입력값
- 회비명
- 금액
- 대상 기간

### 주요 액션
- 저장
- 취소

### 상세 요구사항
- 회비 생성 시 회비 대상 회원에게 납부 상태 레코드를 자동 생성해야 한다.
- 대상 회원 범위는 전체 회원 또는 선택 회원으로 지정 가능해야 한다.
- 이미 납부 기록이 있는 회비는 핵심 값 변경 시 경고해야 한다.

### 검증 규칙
- 금액은 0보다 큰 정수만 허용한다.
- 대상 회원이 0명일 경우 저장할 수 없어야 한다.
- 마감일은 대상 기간과 논리적으로 맞아야 한다.

## 5.7 S-07 회비 납부 현황 화면
### 목적
- 특정 회비 항목의 회원별 납부 상태를 확인하고 관리한다.

### 대상 사용자
- 총무

### 주요 구성 요소
- 회비 기본 정보 요약
- 납부 상태 필터
- 회원별 납부 현황 목록
- 입금 등록 버튼
- 미납만 보기 토글

### 목록 표시 항목
- 회원명
- 입금자명
- 납부 상태
- 입금일
- 입금 금액
- 메모

### 주요 액션
- 상태 필터 적용
- 회원별 상태 수정
- 입금 등록 화면 이동

### 상세 요구사항
- 기본 상태값은 전체 회원 표시로 한다.
- `미납만 보기` 적용 시 미납, 부분 납부, 확인 필요 상태만 보여준다.
- 회원 행에서 바로 상세 수정이 가능해야 한다.

### 상태 정의
- 미납
- 납부 완료
- 부분 납부
- 확인 필요
- 면제

### 예외 처리
- 동일 회비에 중복 납부 등록이 발생하면 경고해야 한다.
- 납부 금액이 회비 금액과 다르면 자동 상태 후보를 제안해야 한다.

## 5.8 S-08 입금 등록/대조 화면
### 목적
- 입금 건을 수동으로 등록하고 예외 상태를 처리한다.

### 대상 사용자
- 총무

### 입력 항목
- 회원 선택
- 회비 항목 선택
- 입금일
- 입금 금액
- 입금자명
- 납부 상태
- 메모

### 필수 입력값
- 회원
- 회비 항목
- 입금일
- 입금 금액

### 주요 액션
- 저장
- 중복 경고 무시 후 저장
- 확인 필요 상태 저장

### 상세 요구사항
- 회비 납부 현황 화면에서 진입한 경우 회비 항목은 자동 선택 상태여야 한다.
- 입금 금액이 회비 금액과 같으면 기본 상태는 `납부 완료`를 제안한다.
- 입금 금액이 더 적으면 `부분 납부`, 더 많거나 판단이 어려우면 `확인 필요`를 제안한다.
- 저장 시 회비 납부 상태와 대시보드 수치가 즉시 갱신되어야 한다.

### 검증 규칙
- 입금 금액은 0보다 커야 한다.
- 미래 날짜 입력은 허용하지 않는다.
- 동일 회원, 동일 회비, 동일 금액, 동일 날짜 조합이 이미 있으면 중복 경고를 표시한다.

### 예외 처리
- 회원을 바로 특정할 수 없는 입금은 임시로 `확인 필요` 상태 저장이 가능해야 한다.

### 목록 표시 항목

- 영수증 누락 여부를 한눈에 확인할 수 있어야 한다.

### 예외 처리
- 마감된 기간의 지출은 기본적으로 읽기 전용으로 보여줘야 한다.

## 5.10 S-10 지출 등록/수정 화면
### 목적

## 6. 기능 명세

## 6.1 F-01 사용자 인증 및 접근 제어
### 설명
- 인증된 총무만 관리 기능에 접근하도록 한다.

### 상세 명세
- 로그인 성공 시 관리 세션을 발급해야 한다.
- 인증되지 않은 요청은 로그인 화면으로 유도해야 한다.
- 서버와 화면 모두 비인증 요청을 차단해야 한다.

### 완료 조건
- 인증 상태에 따라 화면 접근과 API 실행 결과가 일치해야 한다.

## 6.2 F-02 회원 관리
### 설명
- 회비 대상 회원 정보를 생성, 수정, 비활성화한다.

### 상세 명세
- 회원은 이름, 닉네임, 연락처, 입금자명, 회비 대상 여부, 상태, 메모를 가진다.
- 회원 비활성화 시 기존 납부 및 지출 참조는 유지해야 한다.
- 회비 대상 여부가 꺼진 회원은 신규 회비 자동 배정 대상에서 제외한다.

### 완료 조건
- 회원 저장 후 회원 목록과 회비 대상 계산에 즉시 반영되어야 한다.

## 6.3 F-03 회비 항목 관리
### 설명
- 회비 항목을 생성하고 회원별 납부 상태를 초기화한다.

### 상세 명세
- 회비 유형은 월회비, 분기회비, 행사비를 기본 지원한다.
- 회비 생성 시 대상 회원별 납부 레코드를 자동 생성해야 한다.
- 이미 납부 데이터가 있는 회비 항목의 핵심 값 변경 시 경고해야 한다.

### 완료 조건
- 회비 생성 직후 납부 현황 화면에서 회원별 상태가 조회되어야 한다.

## 6.4 F-04 납부 상태 관리
### 설명
- 회원별 회비 납부 상태를 관리한다.

### 상세 명세
- 상태는 `미납`, `납부 완료`, `부분 납부`, `확인 필요`, `면제`를 지원한다.
- 납부 금액과 기준 금액 비교 결과에 따라 기본 상태를 제안해야 한다.
- 총무는 제안 상태를 수동으로 변경할 수 있어야 한다.

### 완료 조건
- 상태 변경 후 회비 항목 요약 수치가 즉시 갱신되어야 한다.

## 6.5 F-05 입금 등록 및 중복 방지
### 설명
- 입금 거래를 등록하고 중복 입력을 줄인다.

### 상세 명세
- 입금 거래는 회원, 회비 항목, 입금일, 금액, 입금자명, 상태를 포함한다.
- 동일 조건 거래가 이미 존재하면 저장 전 경고를 표시해야 한다.
- 입금자를 특정할 수 없으면 `확인 필요` 상태로 저장할 수 있어야 한다.

### 완료 조건
- 입금 저장 후 수입 집계와 잔액 계산이 정확히 반영되어야 한다.

## 6.6 F-06 지출 등록 및 증빙 관리
### 설명
- 지출 거래와 증빙 파일을 관리한다.

### 상세 명세
- 지출은 지출일, 금액, 카테고리, 사용처, 행사/목적, 메모, 첨부 파일을 가진다.
- 첨부 파일은 이미지 및 PDF를 허용한다.
- 지출 수정, 삭제 시 이력을 남겨야 한다.

### 완료 조건
- 지출 저장 또는 수정 이후 정산 데이터와 잔액이 즉시 재계산되어야 한다.

## 6.7 F-07 기초잔액, 이월금, 잔액 계산
### 설명
- 기초잔액과 거래 데이터를 이용해 현재 잔액과 기간별 잔액을 계산한다.

### 계산 규칙
- 초기 기초잔액은 설정 화면에서 1회 이상 설정할 수 있어야 한다.
- 기간 기초잔액 = 직전 마감 기간의 기말잔액, 직전 마감이 없으면 초기 기초잔액
- 기간 수입 합계 = 해당 기간에 확정된 납부 금액 합계
- 기간 지출 합계 = 해당 기간에 확정된 지출 금액 합계
- 기간 기말잔액 = 기간 기초잔액 + 기간 수입 합계 - 기간 지출 합계
- 현재 잔액 = 가장 최근 확정된 기준 잔액 + 이후 거래 반영값
- 삭제된 거래는 합계 계산에서 제외한다.

### 완료 조건
- 대시보드와 정산 화면의 잔액 값이 동일해야 한다.

## 6.8 F-08 정산 및 월 마감
### 설명
- 기간별 회계 요약을 계산하고 월 마감 스냅샷을 저장한다.

### 상세 명세
- 시작일과 종료일을 기준으로 거래를 집계한다.
- 수입, 지출, 순증감, 기말잔액을 표시해야 한다.
- 카테고리별 지출 합계를 제공해야 한다.
- 월 마감 시 스냅샷을 저장하고 해당 기간을 잠금 상태로 처리해야 한다.
- 마감 재오픈은 명시적 액션으로만 가능해야 한다.
- PDF 또는 엑셀 내보내기를 지원할 수 있어야 한다.

### 완료 조건
- 선택한 기간과 동일한 데이터만 정산 결과와 마감 스냅샷에 반영되어야 한다.

## 8. 검증 규칙 요약
- 금액은 정수이며 0보다 커야 한다.

- 필수값 누락 시 저장할 수 없다.
- 첨부 파일은 허용 확장자 및 용량 제한을 만족해야 한다.
- 중복 가능성이 높은 입금 데이터는 경고해야 한다.
- 기초잔액 미설정 상태에서는 정산 확정 또는 월 마감을 할 수 없어야 한다.

## 9. 테스트 관점 체크리스트
- 회원 등록 후 회비 대상 계산이 정상 반영되는가
- 회비 생성 후 회원별 납부 상태가 자동 생성되는가
- 입금 등록 후 대시보드 잔액이 즉시 갱신되는가
- 부분 납부와 면제 상태가 합계에 올바르게 반영되는가
- 지출 수정/삭제 시 정산 수치가 다시 계산되는가
- 기초잔액 설정 후 기간 기초잔액과 현재 잔액이 올바르게 계산되는가
- 월 마감 후 해당 기간 거래가 기본적으로 수정 불가 상태가 되는가
- 변경 이력이 회원, 입금, 지출, 마감에 대해 남는가
